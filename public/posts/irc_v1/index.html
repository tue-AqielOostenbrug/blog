Honoring the IRC Protocol and C-ing it through

2026-03-12

At the start of my master degree I decided that I wanted to do something new. The Computer Science and Engineering bachelor was fun but felt a bit… empty. Although I learned more about how computers work and how to make great programs, at the end of the day it still felt like magic and like Gandalf I also wanted to be able to write my spells from the ground up. So I did the rational thing, and choose Embedded Systems. But the more courses I followed, the greater the gap between the hardware and software felt. So I decided to do the irrational thing… do a Honors project ;)

The project#

To become a proper wizard, I simply felt like I needed some practical experience. No more curriculum, no more constraints, just pure experience. To become a better embedded systems engineer, I decided to get as close to the problem as possible. So I decided to really, really, learn how to program an embeded system in C. Some of the courses I followed already got pretty close to my problem and even increased my curiosity further. The closest of which was Parallelization, compilers and platforms by Roel Jordans. So, I asked for supervision, which he luckily agreed with. Now almost everything was ready, I just needed a concrete-ish plan. So eventually, I came up with the following learning objectives:

  • Program in C
  • Produce a complete embedded product so I can also apply this to industry (design, build, eval, doc, and all)
  • Stay organized (stay Agile)
  • Destroy Isildur’s Bane

An old goody for many new developers is to write a HTTP webserver but I had different plans… During the bachelor there was course called Computer networks and security by Tanir Ozcelebi. While writing a paper on QUIC, I remeber reading about an old chat protocol. A little something called IRC. As a gen-z zoomer this felt like magic from a long forgotten era which I definitely needed to learn some day. And today was that day (I guess). So for my first main objective I decided to tackle IRC. And to go through the whole pipeline I decided to first develop a client for Linux and then port that onto an embedded system. Specifically, an ESP32 after discussing it viability in industy with my supervisor. More so, to add some interesting inputs and actuation I decided to go with the … because it also has a touchscreen (inspired by the ESP32 Marauder by the Just Call Me Koko.)

Program in C#

Ariel, listen to me OO languages? it’s a mess. Programming in C is better than anything they’ve got over there

For both Operating Systems and Parallelization Compilers and Platforms, I already programmed in C. And while doing the project, I even did a special homologation course on it. However, working in a constrained environment for a course was always way easier than to fully create my own. Hence, when I first started playing with sockets in C i was having a lot of trouble. Tons of documentation was available but finding exactly what I needed felt like finding a needle in a haystack. So I first decided to return to the basics… I started reading “The C programming language” by Brain W. Kernighan and Dennis M. Rithchie and my supervisor also let me borrow “LInux Programming het complete hadnboek” by John Goerzen. Then after some practicing with interface and fighting with my biases of Object Oriented Programming with the Return Oriented Programming (… was especially helpful), I could finally get to the juicy part. :D

IRC sockets#

After some advice from my supervisor and Linux Programming het complete Hand Boek, I finally got socket connections to properly work on my computer. Still, for some reason when implementing the protocol like on Wikipedia (…) I simply did not work. There were three big reasons for that:

  1. Not waiting enough
  2. Not ending the line with \r\n
  3. Mistakes in the auth handshake
